Wave 收敛模型

Wave 不是时间盒(Sprint)也不是阶段门(Phase)。它是一个收敛单元：一组可并行执行的任务 + 一个收敛条件。条件满足才结束，不看时间。

知识激活 — 你不需要记任何命令

核心理念

知识存在文件里但不等于被用了。文件是存储，Skill 是激活，Hook 是强制。
你不需要记命令——流程和机制替你记住。你只需要说人话。

三层保障，层层递进

L1 系统自动

Hook 在你开对话的瞬间自动执行：
→ 输出当前 Wave 状态
→ 输出最近新增知识
→ 提醒 Agent 做 preflight/postflight
你什么都不用做，系统保障。

L2 Agent 自觉

CLAUDE.md 规则强制 Agent：
→ 开始任务前：自动跑 preflight（读知识库，注入 pitfall）
→ 完成任务后：自动跑 postflight（捕获知识，验 DoD）
Agent 自己记住，不需要你提醒。

L3 人工兜底

万一 L1+L2 都没拦住，你可以手动：
→ /swarm:preflight T-xxx
→ /swarm:postflight T-xxx
→ /migrate:knowledge query xxx
备用手段，正常不需要。

知识激活的完整链路

你开一个新对话 ↓ [Hook 自动] SessionStart → 输出态势 + 最近知识 + 提醒 ↓ 你说"做 T-065" 或 "继续" ↓ [Agent 自动] preflight → 实时搜索知识库 → 注入相关 pitfall ↓ Agent 写代码... ↓ Agent 写完了 ↓ [Agent 自动] postflight → 引导知识捕获 → 验证 DoD → 更新看板 ↓ [Hook 自动] taskboard 被编辑 → 检测到任务完成 → 提醒沉淀 ↓ 下一个任务... (新知识已经在库里，下次 preflight 自动拿到)

为什么 Skill 每次都实时读？

知识不是静态的——每个任务完成都可能沉淀新 pitfall。
Skill 不缓存，每次执行都从 knowledge/ 目录实时读取最新文件。
5 分钟前另一个 Agent 刚沉淀的 pitfall，下一次 preflight 立刻拿到。

成本：每次 preflight 约读 400 行（INDEX + 匹配文件），约 2k tokens，占对话总量 <1%。完全可忽略。

Core Concept

Knowledge in files is not the same as knowledge being used. Files are storage, Skills are activation, Hooks are enforcement.
You don't need to remember any commands — the process and mechanisms remember for you. Just speak naturally.

Three-Layer Guarantee, Progressive

L1 System Automatic

Hook fires the instant you open a conversation:
→ Output current Wave status
→ Output recently added knowledge
→ Remind Agent to run preflight/postflight
You do nothing. The system handles it.

L2 Agent Discipline

CLAUDE.md rules enforce Agents to:
→ Before task: auto-run preflight (read knowledge base, inject pitfalls)
→ After task: auto-run postflight (capture knowledge, verify DoD)
The Agent remembers. You don't need to remind it.

L3 Manual Fallback

If L1+L2 both missed, you can manually:
→ /swarm:preflight T-xxx
→ /swarm:postflight T-xxx
→ /migrate:knowledge query xxx
Fallback only. Normally not needed.

Complete Knowledge Activation Chain

You open a new conversation ↓ [Hook auto] SessionStart → situation briefing + recent knowledge + reminder ↓ You say "do T-065" or "continue" ↓ [Agent auto] preflight → live search knowledge base → inject relevant pitfalls ↓ Agent writes code... ↓ Agent finishes ↓ [Agent auto] postflight → guides knowledge capture → verifies DoD → updates taskboard ↓ [Hook auto] taskboard edited → task completion detected → deposit reminder ↓ Next task... (new knowledge is already in the base, next preflight picks it up automatically)

Why Does the Skill Always Read in Real Time?

Knowledge is not static — every completed task may deposit a new pitfall.
The Skill does not cache. Every execution reads the latest files from the knowledge/ directory live.
A pitfall deposited by another Agent 5 minutes ago is available at the very next preflight.

Cost: each preflight reads ~400 lines (INDEX + matched files), ~2k tokens, <1% of total conversation volume. Completely negligible.

对抗性仪式 — 大事问鬼

不养常驻红军。在 4 个高杠杆决策点，切换 Agent 为对抗性提示框架。

🔴 事前验尸 Pre-mortem

触发：Spec 签署前
"假设这个 Spec 实现后完全失败了，最可能因为什么？"
5 维度：平台假设 / 数据模型差异 / 用户路径遗漏 / 隐式依赖 / 规模边界

🟠 假设挑战 Assumption Challenge

触发：首次使用新平台 API
"这个 API 行为是文档推测还是真机验证？假设错了影响多大？最小验证 demo 怎么写？"

🟡 魔鬼代言人 Devil's Advocate

触发：Wave 收敛 GO 决策前
"指标都绿，但指标没覆盖什么？修复是治标还是治本？明天发布最丢人的 Bug 是什么？"

🟣 Day-1 灾难推演

触发：发布前
"100 个用户下载后：前 10 秒最可能崩在哪？前 10 分钟最可能丢数据在哪？1 星评论写什么？"

No permanent red team. At 4 high-leverage decision points, switch Agent to an adversarial prompt framework.

🔴 Pre-mortem

Trigger: Before Spec sign-off
"Assume this Spec was implemented and completely failed — what is the most likely cause?"
5 dimensions: platform assumptions / data model differences / missing user paths / implicit dependencies / scale limits

🟠 Assumption Challenge

Trigger: First use of a new platform API
"Is this API's behavior based on docs or device-verified? How bad is it if the assumption is wrong? What is the minimum verification demo?"

🟡 Devil's Advocate

Trigger: Before Wave convergence GO decision
"All metrics are green — but what do the metrics not cover? Is the fix treating symptoms or root cause? What is the most embarrassing bug if we ship tomorrow?"

🟣 Day-1 Disaster Drill

Trigger: Before release
"After 100 users download: where is the most likely crash in the first 10 seconds? Where is data most likely lost in the first 10 minutes? What does the 1-star review say?"

录屏驱动 — 帧帧有据

录屏不只是测试工具，是贯穿全流程的信息源。源码告诉你"代码是什么"，录屏告诉你"用户看到什么"。

阶段录屏怎么用价值
分析行为基线标注 (带时间戳)从录屏提取源码看不到的交互细节
Spec每个 FR 锚定到具体帧 "见 F008"比文字描述精确 10 倍
任务分解帧序列 "从点击到响应经历这 4 帧"AgentH 不用猜
实现先看帧理解目标，再看源码理解实现先知道做成什么样
测试两端录屏逐帧对比 → 差异清单差异量化
验收并排播放，人类终审最直接的"一样不一样"

Screen recordings are not just a testing tool — they are an information source throughout the pipeline. Source code tells you "what the code is"; recordings tell you "what the user sees".
